# Anonymous Question System PRD

- Product Name: `匿名提问与大会答疑系统`
- Version: `V1.0`
- Date: `2026-03-23`
- Form: `H5 网页`

## 1. Background

部门计划召开全员大会，希望在会前收集员工问题，并在会上优先回答大家最关注的问题。现有痛点如下：

- 员工担心被识别，不敢提敏感问题。
- 相同问题重复出现，现场难以排序。
- 如果会前公开全部问题，员工会互相猜测是谁提的。
- 如果允许管理员改写问题，会破坏信任。
- 某些问题不适合会前公开，但仍需要被纳入大会答疑流程。
- 需要兼顾匿名性、组织秩序和会场执行效率，避免内容失控。

## 2. Product Goals

- 提供低门槛、匿名、可信的提问入口。
- 支持 2 种处理方式：`公开问题`、`会上公开`。
- 保证 `原始问题不可编辑、不可改写`。
- 使用 `同类独立提交数` 而不是点赞数衡量问题热度。
- 帮助会务组整理高频议题，并在会上高效展示与答复。

## 3. Non-goals

- 不做会前全员投票。
- 不做评论、回复、辩论区。
- 不做带身份的登录体系。
- 不将该系统作为举报、合规、骚扰、劳动争议等正式处理渠道。
- 不承诺“绝对匿名”或“绝对只有一个人能看到”。

## 4. User Roles

| Role       | Goal                               | Main Actions           |
| ---------- | ---------------------------------- | ---------------------- |
| 员工       | 匿名提问，不被识别，问题被认真处理 | 提交问题，选择处理方式 |
| 会务管理员 | 收集问题、归类议题、安排现场答疑   | 审核、归类、排序、展示 |

## 5. Core Principles

- `原文不可改`：用户提交后的原始问题永久保留，管理员不得编辑或覆盖。
- `议题可归类`：管理员可以创建议题，并将多条相似问题归入同一议题。
- `热度只看独立提交数`：不做点赞，不做支持数。
- `会前默认不公开`：默认处理方式为 `会上公开`。
- `最小可见范围`：谁不需要看，就不应该看。
- `安全优先于完整公开`：即便用户选择公开，若原文含隐私或强识别信息，也可不展示原文，但不可改写原文。

## 6. Core Concepts

| Concept                  | Definition                                            |
| ------------------------ | ----------------------------------------------------- |
| 原始问题 `RawQuestion`   | 用户提交的原文，提交后不可编辑                        |
| 议题 `Topic`             | 会务组创建的归类标签，用于聚合同类问题                |
| 处理方式 `Route`         | `公开问题 / 会上公开`                                 |
| 展示状态 `DisplayStatus` | `原文可展示 / 原文不展示仅计数 / 转正式渠道 / 已答复` |

## 7. Product Form

V1 采用 `H5 网页`，原因如下：

- 更容易做到 `不登录、无身份字段、扫码即提`。
- 比小程序更容易建立“匿名感”。
- 更适合快速搭建员工端入口与大会展示页。
- 研发周期更短，适合先验证机制。

## 8. Route Design

| Route    | Visible Before Meeting   | Visible During Meeting | Counted In Topic Frequency | Who Can See Raw Text By Default | Description                    |
| -------- | ------------------------ | ---------------------- | -------------------------- | ------------------------------- | ------------------------------ |
| 公开问题 | 是，审核后进入公开问题池 | 是                     | 是                         | 员工、会务组                    | 适合愿意提前公开问题的共性问题 |
| 会上公开 | 否                       | 是                     | 是                         | 会务组                          | 默认选项，适合绝大多数问题     |

## 9. User Flow

1. 员工打开匿名提问页。
2. 选择处理方式，默认选中 `会上公开`。
3. 输入问题原文并提交。
4. 系统保存原始问题，不允许后续编辑。
5. `公开问题` 和 `会上公开` 进入会务组处理队列。
6. 会务组归类议题、标记能否展示原文、安排答疑顺序。
7. 大会现场按议题频次和答疑优先级进行展示。
8. 会后发布 FAQ 或未答问题补答结果。

## 10. Page Structure

### 10.1 员工端

- 提问首页
- 公开问题问题池
- 提交成功页
- 风险提示页或正式渠道说明页

### 10.2 会务端

- 问题总览
- 待处理队列
- 议题归类面板
- 大会展示排序页
- 会后 FAQ 导出页

## 11. Employee-side Requirements

### 11.1 匿名提问页

- 输入项：问题原文、处理方式。
- 默认处理方式：`会上公开`。
- 提交前提示：避免填写姓名、团队、项目名、具体时间、可识别个人经历等信息。
- 只支持文本提交，V1 不支持图片、附件、语音。
- 文本长度限制建议为 `10-500 字`。

### 11.2 公开问题问题池

- 仅展示已审核通过的公开问题内容。
- 不提供点赞、支持、评论功能。
- 员工可查看问题原文、所属议题、答复状态。
- 员工仍需通过提问页重新提交自己的问题，不做一键支持。

### 11.3 提交成功页

- `公开问题`：提示“问题将在审核后进入公开问题池”。
- `会上公开`：提示“会前不公开，将在大会现场统一展示安排”。

## 12. Organizer-side Requirements

### 12.1 问题查看

- 查看 `公开问题` 与 `会上公开` 的原始问题。
- 按时间、处理方式、状态、议题筛选。
- 不允许编辑原文内容。

### 12.2 议题归类

- 支持创建议题。
- 支持将多条原始问题挂到同一议题下。
- 支持查看某议题下的独立提交数量。
- 议题标题由会务组创建，用于归类，不替代原始问题。

### 12.3 展示控制

- 每条问题可标记为：
  - `原文可展示`
  - `原文不展示，仅计入议题频次`
  - `转正式渠道`
  - `已答复`
- 若原文包含隐私、点名、攻击性或强识别信息，可不展示原文，但不可改写原文。
- 原文不可展示时，大屏可仅展示议题标题和频次。

### 12.4 排序规则

- 默认按 `议题下独立提交数` 降序排序。
- 支持手动调整大会展示顺序。
- 支持少量“管理层指定必答议题”置顶。
- 不支持按点赞、浏览量、评论量排序。

## 13. Meeting Board Requirements

- 展示议题标题、同类提交数、答复状态。
- 可选择展示该议题下 `1-3 条可公开原文`。
- 对于无可公开原文的议题，仅展示议题标题与频次。
- 支持“大屏模式”和“主持人控制模式”。
- 支持现场标记 `已回答 / 待会后补答`。

## 15. Permission Matrix

| Capability       | 员工 | 会务管理员 |
| ---------------- | ---- | ---------- |
| 提交问题         | 有   | 无         |
| 查看公开问题池   | 有   | 有         |
| 查看会上公开原文 | 无   | 有         |
| 编辑原文         | 无   | 无         |
| 创建或管理议题   | 无   | 有         |
| 调整大会排序     | 无   | 有         |

## 17. Security And Risk Control

- 不接 SSO，不收集姓名、工号、邮箱、OpenID。
- 不嵌入第三方统计、热图、录屏脚本。
- 接口和网关不得记录明文请求体。
- 对外只承诺“尽量减少身份信息采集”，不承诺绝对匿名。
- `公开问题` 和 `会上公开` 页面均提示用户避免填写可识别信息。
- 涉及举报、歧视、骚扰、舞弊、劳动争议等内容，页面需引导至正式渠道。
- 公开路由文本需经过审核后才能展示。
- V1 需具备基础防刷能力：限流、验证码或轻量风控，但不引入实名。

## 18. Review Rules

- 可展示：共性问题、制度问题、流程问题、资源问题、方向问题。
- 不展示原文但可计入议题：包含姓名、团队名、具体项目、具体时间、强识别经历、情绪化攻击。
- 转正式渠道：举报、合规、骚扰、歧视、个人权益争议等。
- 会务组不得修改原文，不得以“润色”为名替换提问内容。

## 19. Key Copy Principles

- 默认处理方式为 `会上公开`。
- 页面明确提示“会前默认不公开”。
- 页面明确提示“原始问题不会被改写，但不代表所有原文都会被公开展示”。

## 20. Success Metrics

- 提交完成率
- `会上公开` 占比
- 高频议题覆盖率：大会答复议题数 / Top 议题总数
- 会后补答完成率
- 公开展示安全率：公开内容零隐私事故
- 用户对匿名可信度的调研评分

## 21. Acceptance Criteria

- 用户可在 1 分钟内完成匿名提交。
- 会务管理员无法编辑任何原始问题。
- 大会排序依据可按同类独立提交数自动生成。
- 会前默认不公开 `会上公开` 类问题。
- 大屏可仅展示议题，不强制展示原文。

## 22. In Scope For V1

- H5 提问页
- 公开问题池
- 会务管理台
- 大会展示页
- FAQ 导出

## 23. Out Of Scope For V1

- 小程序版本
- 评论区
- 点赞或支持
- 附件上传
- 智能自动归类
- 多语言
- 组织架构维度报表

## 24. Open Questions

- 原始问题与审计日志保留多久。
- 正式渠道由谁承接，页面跳转到哪里。
- `公开问题` 问题池是否对全员开放，还是仅限会务二维码入口访问。
