# Frontend Wireframes

本文档用于描述 V1 的页面结构、布局重点、交互方式和响应式建议，供设计和前端实现参考。

## 1. Information Architecture

```text
员工端 H5
├─ 匿名提问页
├─ 提交确认弹窗
├─ 提交成功页
└─ 公开讨论池

会务端 Web
├─ 问题管理页
├─ 议题管理页
└─ 大会展示控制页

领导端 Web
└─ 保密查看页
```

## 2. Employee H5

### 2.1 匿名提问页

目标：

- 让用户在 1 分钟内完成提交。
- 清楚理解三种处理方式的区别。
- 建立“这个系统尽量匿名、不会改我原话”的信任感。

Wireframe:

```text
┌──────────────────────┐
│ 顶部标题：匿名提问       │
│ 副标题：选择处理方式并提交 │
├──────────────────────┤
│ 处理方式               │
│ ○ 公开讨论            │
│   提交后进入公开问题池   │
│ ○ 会上公开（默认）      │
│   会前不公开，会上展示   │
│ ○ 直达领导            │
│   不上会，仅指定收件人看 │
├──────────────────────┤
│ 问题输入框             │
│ [ 多行文本区域        ] │
│ [ 占位提示文案        ] │
│ 字数：32 / 500         │
├──────────────────────┤
│ 风险提示区             │
│ 不收集姓名工号         │
│ 原始问题不会被改写      │
│ 避免填写可识别细节      │
├──────────────────────┤
│ [ 匿名提交 ] 按钮       │
└──────────────────────┘
```

Layout Notes:

- 顶部是简短标题和说明，不用大段说明文。
- `处理方式` 用 3 张可点击卡片，不建议用下拉框。
- 默认高亮 `会上公开`。
- 输入框放在处理方式下方，形成“先选路由，再写内容”的心智。
- 风险提示固定放在提交按钮上方。

Interaction Notes:

- 点击处理方式卡片，卡片边框和背景变化。
- 输入框下方实时显示字数。
- 未满足最小字数时，提交按钮置灰。
- 页面不要出现任何“登录后查看更多”的暗示。

Responsive Notes:

- 单列布局。
- 主按钮吸底或固定底部。
- 卡片圆角、边框清晰，少做花哨动效。

### 2.2 提交确认弹窗

目标：

- 让用户在最后一步明确知道自己的问题会如何流转。

Wireframe:

```text
┌──────────────────────┐
│ 确认提交              │
├──────────────────────┤
│ 你选择的是：会上公开    │
│ 会前不公开，大会现场展示 │
│                      │
│ 你的原始问题不会被改写  │
│ 请再次确认是否提交      │
├──────────────────────┤
│ [ 取消 ]  [ 确认提交 ]  │
└──────────────────────┘
```

Notes:

- 三种处理方式对应三套文案，但结构保持一致。
- 弹窗聚焦“流转后果”，不要塞太多解释。
- `直达领导` 需要额外强调“不上会”。

### 2.3 提交成功页

目标：

- 给用户一个明确闭环，降低“提交后黑箱感”。

Wireframe:

```text
┌──────────────────────┐
│ 提交成功              │
├──────────────────────┤
│ 你的问题已提交         │
│ 会前不会公开           │
│ 将视安排在会上展示      │
├──────────────────────┤
│ [ 继续提问 ]           │
│ [ 查看公开讨论 ]       │
└──────────────────────┘
```

Notes:

- 文案根据处理方式切换。
- `直达领导` 成功页不展示“查看状态”入口。

### 2.4 公开讨论池

目标：

- 仅展示已审核通过的公开问题。
- 让用户感受到系统会真实处理问题。

Wireframe:

```text
┌──────────────────────┐
│ 顶部标题：公开讨论      │
│ 说明：仅展示审核通过内容 │
├──────────────────────┤
│ [ 搜索框 ]            │
│ [ 议题筛选 ]          │
├──────────────────────┤
│ 卡片 1               │
│ 问题原文……            │
│ 议题：晋升标准         │
│ 状态：待回应           │
│ 日期：2026-03-23      │
├──────────────────────┤
│ 卡片 2               │
│ ……                   │
└──────────────────────┘
```

Notes:

- 不要点赞按钮，不要评论，不要支持。
- 日期只显示到天，不显示具体时间。
- 公开讨论池和匿名提问页之间要能互相跳转。

Empty State:

```text
暂无公开问题
你仍然可以先匿名提交自己的问题
[ 去提问 ]
```

## 3. Organizer Web

### 3.1 问题管理页

目标：

- 处理 `公开讨论` 和 `会上公开` 两类问题。
- 重点是看原文、归类、改状态，但不能改字。

Wireframe:

```text
┌───────────────────────────────────────────────┐
│ 顶部导航：问题管理 | 议题管理 | 大会展示       │
├───────────────────────────────────────────────┤
│ 筛选栏                                        │
│ [处理方式] [展示状态] [议题] [时间范围] [搜索] │
├───────────────────────────────────────────────┤
│ 表格                                          │
│ ID | 处理方式 | 原始问题 | 议题 | 展示状态 | 时间 │
│------------------------------------------------│
│ 01 | 会上公开 | ……       | 未归类 | 待处理   | …  │
│ 02 | 公开讨论 | ……       | 晋升   | 可展示   | …  │
├───────────────────────────────────────────────┤
│ 右侧详情抽屉 / 弹层                            │
│ 原始问题全文                                   │
│ 处理方式：会上公开                             │
│ 当前议题：未归类                               │
│ 展示状态：待处理                               │
│                                                │
│ [归入议题] [标记原文可展示] [仅计数不展示]     │
│ [转正式渠道] [标记已答复]                      │
└───────────────────────────────────────────────┘
```

Layout Notes:

- 左侧主列表，右侧详情抽屉是推荐结构。
- 原始问题在详情里完整显示。
- 页面顶部固定一句提示：`原始问题不可编辑、不可改写。`

Interaction Notes:

- 点击列表行，右侧抽屉展开。
- 所有操作都是“状态变更”或“归类”，没有编辑按钮。
- 状态变更后，列表实时更新。

### 3.2 议题管理页

目标：

- 聚合同类问题，形成大会可用议程。

Wireframe:

```text
┌─────────────────────────────────────────────┐
│ 顶部标题：议题管理                           │
├─────────────────────────────────────────────┤
│ [ 新建议题 ] [ 按问题数排序 ] [ 导出 FAQ ]    │
├─────────────────────────────────────────────┤
│ 议题列表                                     │
│ 议题标题 | 同类问题数 | 可展示原文数 | 排序 | 状态 │
│------------------------------------------------│
│ 晋升标准透明度 | 18 | 4 | 1 | 待答复         │
│ 绩效校准机制   | 12 | 3 | 2 | 待答复         │
├─────────────────────────────────────────────┤
│ 点击议题后侧边栏                              │
│ 议题标题                                      │
│ 同类问题列表                                  │
│ [问题1 原文]                                  │
│ [问题2 原文]                                  │
│ [问题3 原文]                                  │
│                                                │
│ [ 调整排序 ] [ 标记已答复 ]                    │
└─────────────────────────────────────────────┘
```

Notes:

- 议题视角优先，而不是单题视角。
- `同类问题数` 是最重要字段，视觉上应突出。
- 排序可以用拖拽或上下移动按钮，选其一。

### 3.3 大会展示控制页

目标：

- 提供主持人和会务的现场控制台。

Wireframe:

```text
┌─────────────────────────────────────────────┐
│ 左侧：题目列表                                │
│ 1. 晋升标准透明度 18                          │
│ 2. 绩效校准机制   12                          │
│ 3. 跨团队协作     10                          │
├─────────────────────────────────────────────┤
│ 右侧：当前展示内容                             │
│ 议题标题：晋升标准透明度                       │
│ 共收到 18 条相近问题                           │
│                                                │
│ 原文 1：……                                     │
│ 原文 2：……                                     │
│ 原文 3：……                                     │
│                                                │
│ 状态：待回答                                   │
│ [ 上一题 ] [ 下一题 ] [ 标记已回答 ] [ 会后补答 ] │
└─────────────────────────────────────────────┘
```

Notes:

- 左侧是议题队列，右侧是当前题卡。
- 当前题卡应适配投屏比例，内容尽量少而大。
- 若某议题无可展示原文，则只展示议题标题和频次。

## 4. Leader Web

### 4.1 保密查看页

目标：

- 只处理 `直达领导` 消息。
- 页面气质要克制、可信，不像普通后台。

Wireframe:

```text
┌──────────────────────────────────────┐
│ 顶部标题：保密查看                     │
│ 说明：消息为加密存储，需本地解密查看     │
├──────────────────────────────────────┤
│ [ 导入私钥 ]                           │
│ 当前状态：未导入 / 已导入               │
├──────────────────────────────────────┤
│ 消息列表                               │
│ 2026-03-23 上午  新消息                │
│ 2026-03-23 上午  已读                  │
├──────────────────────────────────────┤
│ 右侧或下方阅读区                        │
│ [ 点击后本地解密显示正文 ]              │
│                                        │
│ [ 标记已读 ] [ 归档 ]                   │
└──────────────────────────────────────┘
```

Notes:

- 先导入私钥，再解密查看。
- 未导入私钥时，正文区只提示“需导入私钥后查看”。
- 不显示复杂后台导航，避免与会务系统混淆。

## 5. Navigation Map

```text
匿名提问页
├─ 提交确认弹窗
├─ 提交成功页
└─ 公开讨论池

问题管理页
├─ 议题管理页
└─ 大会展示控制页

保密查看页
└─ 消息详情解密视图
```

## 6. Key UI States

### 6.1 Employee-side

- 默认态：`会上公开` 已选中，提交按钮灰置。
- 可提交态：字数满足要求，按钮高亮。
- 提交中：按钮 loading，不可重复提交。
- 成功态：显示结果页。
- 失败态：顶部 toast，保留已输入内容。

### 6.2 Organizer-side

- 问题状态：`待处理 / 原文可展示 / 仅计数不展示 / 转正式渠道 / 已答复`
- 议题状态：`待答复 / 已回答 / 会后补答`

### 6.3 Leader-side

- 私钥未导入
- 私钥已导入
- 消息未读
- 消息已读
- 消息已归档

## 7. Responsive Guidance

- 员工端严格按移动端优先设计，宽度以 `375px` 基准稿为主。
- 会务端最低适配 `1280px` 宽屏。
- 大会展示页优先适配 `16:9` 投屏。
- 领导端优先适配桌面端，不必优先做手机体验。

## 8. Design Reminders

- 员工端第一重点不是“功能丰富”，而是“可信、克制、没有追踪感”。
- `会上公开` 必须始终是默认选项和主推荐项。
- `公开讨论` 不要设计得过于强势，否则会削弱匿名感。
- 会务端最重要的视觉提示是：`原始问题不可编辑、不可改写。`
- 领导端最好让人一眼看出它是独立保密通道，而不是普通后台页面。
